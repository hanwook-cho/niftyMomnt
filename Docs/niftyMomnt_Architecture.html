1. Purpose of This Document

This Architecture Decision Record captures the why behind every major structural decision in niftyMomnt — illustrated visually so that any engineer, designer, or collaborator can understand the system at a glance before reading the detailed SRS.

The SRS (v1.2) specifies what each component does and how its interfaces are defined. Product behavior and scope come from the PRD (v1.6); screen-level design from the UI/UX Specification (v1.7). This ADR answers why the system is shaped the way it is, and shows how the pieces fit together through diagrams that cannot be embedded in a Word document.

Core requirement that shaped everything: the UI layer must be completely substitutable without touching any business logic, data access, or AI processing code. Multiple app variants — different features, different UI — must share a single codebase except for their UI and configuration.

2. Layered Architecture — The Four-Layer Model

2.1 Overview

niftyMomnt uses Clean Architecture adapted for iOS. The system is divided into four layers. The single most important rule is:

Dependencies point inward only. Outer layers know about inner layers. Inner layers know nothing about outer layers. The domain layer — all business logic — has zero knowledge of UIKit, SwiftUI, or any Apple platform framework.

This rule is what makes the UI swappable. If the domain layer never imports SwiftUI, swapping the entire UI layer requires zero changes to any engine or use case.

Figure 1 — Four-layer Clean Architecture. Dependencies flow inward only. The domain layer (amber) contains all business logic and has zero platform imports. Swapping the entire UI layer (green) requires zero changes to any amber or red layer.

2.2 Why This Structure — The Decision

The alternative was a more conventional layered iOS app where ViewModels directly call services that use platform frameworks. That approach would work fine for a single-variant app. But niftyMomnt needs to support multiple app variants with different feature sets and different UIs — and the capture engine, indexing engine, and AI pipeline must be identical across all of them.

Rejected approach

ViewModel → Service → AVFoundation directly. Fast to build initially. Becomes a problem when you need a Lite variant: you either duplicate the service code or start adding conditional logic everywhere. UI and platform concerns become entangled.

Chosen approach

ViewModel → Protocol (domain) → Adapter (data) → Platform. Adding a Lite variant means: new Xcode target + new UI + new AppConfig. The capture engine, indexing engine, and all business logic are touched zero times.

2.3 The Dependency Rule in Practice

The rule is enforced by CI, not by convention. A SwiftLint custom rule scans every file in NiftyCore/Sources/ and fails the build if it finds any import of UIKit, SwiftUI, AVFoundation, CoreML, or any Apple platform framework. This makes the rule impossible to accidentally violate.

3. Multi-Variant Architecture — AppConfig Model

3.1 The Problem It Solves

You have described wanting several variations of niftyMomnt: a full-featured version with the MZ aesthetic UI, a simpler version with fewer features, and potentially other configurations. All of these need to share the same capture engine, the same AI pipeline, the same vault, and the same intelligence graph. Only the UI and the feature configuration should differ.

Adding a new app variant = new Xcode target + new UI layer + new AppConfig. Zero lines of NiftyCore are modified. The same CaptureEngine, IndexingEngine, StoryEngine, NudgeEngine, VaultManager, and GraphManager serve every variant identically.
Figure 2 — Multi-variant architecture. Three app targets (top, purple) each have their own UI layer (green) and AppConfig. All share identical NiftyCore engines (amber, bottom). Adding a new variant never touches NiftyCore code.

3.2 How AppConfig Works

AppConfig is a Swift value type — a struct — injected into every engine at startup. It is an OptionSet-based capability declaration. The engines read it to decide which modes are available, which AI features are active, and which sharing capabilities are enabled.

3.3 What "New Variant" Looks Like in Practice

Suppose you want a travel-agency branded version with a white-label UI and only Still + Atmosphere capture:

  1. Create a new Xcode target: niftyMomntBrand
  2. Create AppConfig+Brand.swift with assetTypes: [.still, .atmosphere] and the desired feature set
  3. Create the UI layer in Apps/niftyMomntBrand/UI/ — SwiftUI views with the brand's design system
  4. Wire the composition root in niftyMomntBrandApp.swift

Zero files in NiftyCore are opened, let alone modified. The capture engine, indexing engine, vault, and intelligence graph are inherited without a single line of change.

4. Protocol Boundaries — How Layers Talk

4.1 The Pattern

Every dependency that crosses a layer boundary is expressed as a Swift protocol defined in the domain layer. The domain layer owns the contract. The data layer provides the implementation. The presentation layer and engines depend only on the contract — never the implementation.

This is what makes each layer independently testable. To test CaptureEngine in isolation, inject a MockCaptureAdapter that conforms to CaptureEngineProtocol. The test never touches AVFoundation, never requires a physical device, and runs in under a millisecond.
Figure 3 — Protocol boundary pattern. The ViewModel calls the protocol (domain). CaptureEngine implements it without importing AVFoundation. AVCaptureAdapter (data layer) wraps the real hardware. Tests inject MockCaptureAdapter — no hardware, no network, instant execution.

6. Companion App Data Layer

6.1 Two-Layer Separation

The intelligence graph solves the companion app data problem elegantly. Instead of sharing 50–200GB of raw assets across apps, only the metadata graph — typically under 50MB — is shared. Raw assets stay locked in the vault.

Figure 5 — Companion app data access model. The intelligence graph (amber, <50MB) is readable by all companion apps via the shared App Group container. Raw vault assets (red, 50–200GB) stay locked in the niftyMomnt sandbox — companion apps request individual assets via XPC with per-request user consent.

7. Key Architectural Decisions — Summary

#DecisionWhat was decidedWhy
ADR-01 Clean Architecture (4 layers) App Target → Presentation → Domain → Data, with inward-only dependencies Enables UI substitution without touching business logic. Enforced by CI SwiftLint rule.
ADR-02 NiftyCore as a local Swift Package All business logic in a separate Swift Package — not inside any app target Makes shared code physically impossible to accidentally entangle with UI code. Package boundary is enforced by the Swift build system.
ADR-03 AppConfig OptionSet for variants Capability flags injected at startup, checked by each engine at runtime New variant = new Xcode target + new AppConfig. Zero engine code changes. All variants always run the same business logic — just with different capability sets.
ADR-04 Protocol-defined cross-layer contracts Every cross-layer dependency is a protocol in the domain layer Makes every engine unit-testable in isolation with mock implementations. No physical hardware, no file system, no network required in engine tests.
ADR-05 Swift Concurrency only (no GCD) async/await + actors + structured concurrency throughout Actors provide compile-time data-race safety. Structured concurrency makes cancellation predictable. No manual queue management.
ADR-06 Intelligence graph separate from vault Raw assets in AES-256 vault (sandbox only). Metadata in SQLCipher graph (App Group shared container) Companion apps need metadata (<50MB), not raw assets (50–200GB). Separation keeps the sharing surface small and the privacy surface manageable.
ADR-07 Three-tier AI processing model Mode 0 on-device (default) / Mode 1 text-only Enhanced AI / Mode 2 Lab encrypted visual — separate toggles, separate data contracts Resolves the contradiction between "never send photos" and "server AI processes images." Each mode has a precisely accurate privacy statement. No ambiguity.
ADR-08 Zero platform imports in domain layer CI enforces that no file in NiftyCore imports UIKit, SwiftUI, AVFoundation, CoreML, or any Apple framework The domain layer is purely about business logic. Platform specifics belong in adapters. This makes NiftyCore portable — theoretically runnable on any Swift platform.
