1. Overview & Scope

1.1 Purpose of This Document

This Product Requirements Document (PRD) defines the functional behavior, UX logic, technical architecture, and feature acceptance criteria for niftyMomnt v1.0. It is the primary reference for design, engineering, and QA. Screen-level layout and interaction design are specified in the companion UI/UX Specification v1.7. Market and user research are documented in MRD v1.5 (same revision family as the cover line “Builds on MRD v1.5”).

1.2 Product Summary

niftyMomnt is a moments studio for iOS. It captures five distinct asset types through a unified Capture Hub, applies on-device intelligence to tag and cluster moments automatically, and enables users to shape those moments into stories — for public sharing, trusted-circle sharing, or private keeping.

Design principle: Every interaction should feel like the app understood what you wanted before you did.

1.4 Design Constraints

  • Platform: iOS 26.0 minimum. iPhone 15 and above. Required for updated AI Intelligence frameworks, newly updated SDK APIs, AVFoundation AVCaptureMultiCamSession, Neural Engine performance, Journaling Suggestions API, and computational photography APIs. No iPadOS in v1.0. UPDATED v1.5
  • Offline-first — hard requirement: The app must be fully functional with zero network connectivity. This includes capture of all 5 asset types, on-device indexing, Film Archive browsing, Reel assembly, preset rendering, Sound Stamp acoustic analysis, and AI Nudge Engine. The only features requiring connectivity are Lab cloud processing (opt-in) and trusted-circle link delivery.
  • Privacy non-negotiable: Private vault content must be end-to-end encrypted. Never used for AI training. Never surfaced in community feed. Never watermarked. Dual-camera secondary capture and Sound Stamp audio are processed on-device only and never stored as viewable or playable files.
  • Private by design — brand claim: niftyMomnt contains no third-party trackers, no third-party analytics SDKs, and no advertising networks.
  • On-device first: All high-frequency operations run on-device. Cloud ("Lab") is opt-in only, ephemeral, and explicitly user-triggered.
  • Performance target: Capture-to-preview latency under 300ms. Cold launch under 1.5s. Film Archive load under 2s for up to 5,000 assets. Sound Stamp acoustic analysis completes within 300ms of buffer end — does not block UI.

2. Asset Taxonomy — The Five Moment Types + Life Four Cuts

2.1 Overview

niftyMomnt defines five discrete asset types — each designed to capture a different dimension of a moment. Together they form a complete sensory vocabulary: image, motion, time, sound, and atmosphere. Every asset type feeds the same downstream pipeline: on-device indexing, vibe tagging, Film archive organization, and story assembly.

Design principle: The asset type should feel like a natural extension of what the moment calls for — not a mode the user has to consciously choose.

Non-Destructive Derivative Assets NEW v1.5

The Fix feature (§3.8) introduces a derivative asset state for Still and Live assets. When a user applies a Fix (crop, rotate, or flip), the system writes a derived version alongside the original in the vault. The original is never modified or deleted. Key rules governing derivatives:

  • The derivative is the primary displayed version in the Film Archive, Moment detail, widgets, and share flows.
  • The original is retained silently — always recoverable via "Revert to original."
  • Derivatives are stored with a fix_applied: true flag and a reference to the source asset ID in the intelligence graph.
  • Lab processing and Enhanced AI always operate on the derivative if one exists — never on the original unless the derivative has been reverted.
  • Derivatives inherit all vibe tags, acoustic tags, and ambient metadata from the original asset. Fix does not re-trigger indexing.
  • A derivative does not count as a new asset for Roll Mode or streak calculations.

2.4 Asset Interoperability & Export

  • Still: Exports as HEIF (default) or JPEG. When a derivative exists (Fix applied), the derivative is exported by default. Watermark applied on free tier.
  • Live: Exports as Apple Live Photo (HEIF + MOV pair). Fix applies to the HEIF component only — MOV exports at original dimensions. Falls back to Still in non-Apple contexts.
  • Clip: Exports as MP4 (H.264). Compatible with Instagram Reels, TikTok, and all major platforms.
  • Echo: Exports as M4A audio file. Voice-to-Prose engine generates three rewrite variants from transcript.
  • Atmosphere: Exports as MP4 (still + audio looped) for social sharing. Alternatively as HEIF + M4A pair for archive.

Aspect ratio note: Capture remains non-destructive at the sensor/original asset level. The selected ratio acts as a framing and export target for Still and Live assets, with 9:16 as the default social-first composition.

2.5 Dual-Camera Contextual Capture

Privacy principle: The secondary camera image is processed entirely on-device, never stored as a viewable file in the vault, and never transmitted under Mode 0 (on-device) or Mode 1 (Enhanced AI text-only). It is metadata, not an asset. The user is always the owner of what the secondary camera captures, and can disable this feature in Settings → Capture.

When the user captures a Still or Live asset, the secondary camera (whichever camera is not currently primary) fires simultaneously. The secondary image is never shown to the user, never stored as a viewable photo, and never exported. It is processed immediately on-device to extract contextual metadata, then discarded. This makes every Still and Live capture automatically richer — without adding any UI complexity or user decision.

The system is bidirectional — it adapts based on which camera the user has selected as primary:

Nudge questions reference social context. Annual Review: e.g. "You were alone in 38% of your moments this year." Mood Map weights social context. Intelligence graph: social_context field on Asset.

Moment label becomes: "Selfie · Golden hour · coastal cliff" instead of just "Selfie." Chromatic palette from the scene contributes to Mood Map coloring. Film card ambient metadata line gains location context.

Secondary capture — technical behavior

  • Capture timing: Secondary camera fires within 50ms of primary shutter. Both captures share the same timestamp. AVFoundation multi-camera session (AVCaptureMultiCamSession) requires supported hardware; v1.6 minimum device is iPhone 15 (see §13.4).
  • Processing pipeline: Secondary image is handed directly to the IndexingEngine as a transient buffer — never written to disk, never passed to the VaultRepository. Classification results are written to the Asset metadata record and the buffer is released. Total on-device processing time: under 300ms.
  • Privacy controls: User can disable secondary camera capture entirely in Settings → Capture → "Contextual context capture." When disabled, Still and Live captures behave as before — no secondary metadata enrichment. Default: on.
  • Lab (Mode 2) behavior: When the user initiates a Lab session for a Still or Live asset, the consent modal explicitly states whether the secondary context metadata (person count, scene type) will be included. User can exclude secondary metadata from Lab sessions via a toggle in the consent screen. Default: excluded.
  • Self-timer interaction: When self-timer is active (3s or 10s), both cameras fire simultaneously at the end of the countdown. The rear camera becomes the secondary capture for selfie-oriented self-timer shots — providing the scene context for what was behind the user during the timer.

2.6 Sound Stamp — Ambient Audio for Still Captures

Privacy principle: Sound Stamp audio is never written to disk under any mode. The 1.5s buffer exists only in memory, passes through the on-device acoustic classifier (same pipeline as §4.3), and is released. The audio is metadata infrastructure, not an asset. It is never stored, never exported, and never transmitted — including under Mode 1 (Enhanced AI) and Mode 2 (Lab).

2.6.1 Overview

Sound Stamp is an opt-in feature that captures a 1.5-second ambient audio buffer during every Still capture — 0.5s pre-roll from an always-on microphone buffer (same buffering model as Live Photo) plus 1.0s post-shutter. The buffer is analysed on-device for acoustic vibe tags identical to those defined in §4.3, then immediately discarded. No audio file is ever created.

Sound Stamp closes the acoustic tagging gap for Still-only users. Prior to v1.5, §4.3 Acoustic Signature Analysis applied only to Echo, Clip, and Atmosphere assets. A user who primarily captures Stills received zero acoustic vibe enrichment. Sound Stamp extends acoustic tagging to every Still capture without requiring mode switching or user decision.

2.6.4 Permission & Opt-In Model

  • Default: Off. Sound Stamp is never active on first install without explicit user action.
  • Onboarding: Offered in Step 5 (Optional enrichments) as "Sound Stamp" with plain-language explanation: "Capture 1.5 seconds of ambient sound with every photo — niftyMomnt uses it to tag the vibe and immediately discards the audio. Nothing is stored or shared."
  • Settings → Capture → Sound Stamp: Toggle (default: Off). Sub-label: "1.5s ambient audio at capture. Analysed on-device, never stored." Expandable "Learn more" explains what is and isn't retained.
  • Microphone permission: NSMicrophoneUsageDescription is already requested in Onboarding Step 3 for Echo and Atmosphere capture. Sound Stamp does not require a new permission — it uses the existing grant. If microphone permission is denied, Sound Stamp is silently unavailable (toggle is greyed out in Settings with a note: "Requires microphone access — enable in iOS Settings.").

2.6.5 Naming Discipline

"Sound Stamp" is the only permitted name for this feature in all user-facing surfaces. Never use "ambient recording," "microphone capture," "audio sample," or similar phrasing that implies stored content. The name signals brevity and output type — a stamp is a quick impression, not a file.

The Capture Hub is the primary screen and heart of niftyMomnt. The UI disappears into the moment — controls surface only when needed. All overlay surfaces use iOS 26 Liquid Glass materials (UIVisualEffectView with .systemUltraThinMaterialDark).

This ensures the Nudge Engine always has access to the most recent user-provided vibe signal, making its reflection prompts more contextually accurate.

7. Story Engine & AI Reel Assembler

7.1 Overview

The Story Engine transforms raw Moment clusters into publishable content. Its primary output is the AI Reel — an automatically assembled short-form video (15–60s) ready to post to Instagram Reels or TikTok. Secondary outputs include photo carousels, trip journals, and private memory compilations. The Story Engine draws on both Cold indexing metadata and (optionally) Lab-enriched narrative graphs. Fix-corrected assets (derivatives) are used in Reel assembly and Film Archive hero selection — the system always works from the primary displayed version of each asset. Sound Stamp acoustic tags from Still assets contribute to Reel sequencing logic and music matching in the same way Echo acoustic tags do.

7.2 AI Reel Assembler — Logic

Input Selection

  • Source: All Clips, Stills, Lives, and Atmospheres within a selected Moment cluster or date range.
  • Asset scoring: Each asset receives a score based on: motion interest (Clips with subject movement score higher), vibe coherence (assets matching the dominant Moment vibe score higher), chromatic harmony (assets whose palette complements the cluster palette score higher), and uniqueness (avoids selecting near-identical Stills from burst sequences).
  • Target count: Assembler selects 8–15 assets for a 15–45s Reel. User can adjust count before render.

Sequencing Logic

  • Opening: The highest-scoring wide/context shot or an Atmosphere asset (most immersive opening).
  • Arc: Mid-section assets sequenced by time (default) or vibe similarity (alternative, user-selectable). Clips are distributed throughout — no back-to-back Clip stacking unless the user enables it.
  • Close: The highest-scoring "emotionally resonant" asset — determined by vibe tag: a Sunset, a Quiet moment, or a Crowd asset with high energy.
  • Lab enhancement: If Lab narrative graph is available, sequencing follows the narrative arc (opening beat → arc beat → closing beat) rather than time or vibe heuristics.

Music & Transitions

  • Music: Curated royalty-free tracks organized by vibe tag. System selects a track matching the dominant Moment vibe. User can swap from a preset-matched library. No copyrighted music suggested by default (flags Echo assets tagged "Music" as potentially rights-sensitive on public export).
  • Transitions: Cut (default), cross-dissolve, and motion blur available. Transition style is tied to the active vibe preset family — lo-fi presets use hard cuts; editorial presets use cross-dissolves.
  • Voice note overlay: If one or more Echo assets exist within the Moment cluster, their transcribed text (or Voice-to-Prose "Minimalist" variant) can be overlaid as animated text on the Reel. The overlay is optional — user toggles it before render. Typography style matches the active vibe preset.

Duration match: Clip durations are trimmed to fit the music's phrase boundaries where possible (beat-matched at 1-bar intervals).

8. AI Nudge Engine — Post-Capture Reflection

Nudge question tone and chip copy are updated for the MZ audience per UI/UX Spec v1.7 §6.3 — questions are phrased in aesthetic/felt register, not analytical register. Sound Stamp acoustic tags from Still assets feed the Nudge Engine trigger logic (Acoustic signal trigger pattern) in the same way Echo acoustic tags do.

8.1 Overview

The AI Nudge Engine is niftyMomnt's primary retention driver between active capture sessions. It proactively surfaces a contextual in-app card a few minutes after the user stops moving or capturing — asking a single, intelligent question about what they just experienced before the emotional memory fades. The interaction is designed to take under 15 seconds and feel like a thoughtful friend asking the right question, not a journal prompt.

Design principle: One question. Two or three taps. Done. The nudge captures emotional context in the moment of highest emotional availability — not hours later when the feeling is gone.

8.2 Post-Capture Trigger Sequencing UPDATED v1.6

The AI Nudge Engine does not fire during the post-capture overlay window. The confirmed sequencing is:

  1. Capture event fires → Post-Capture Overlay opens (§3.9)
  2. Vibe Tag Prompt is presented (T+300ms, 4-second window)
  3. User-selected vibe tags are written to the asset's metadata record in the intelligence graph
  4. Post-Capture Overlay closes (T+7.5s or on user dismissal)
  5. AI Nudge Engine fires, reads the intelligence graph including any user-provided vibe tags from step 3, and generates a reflection card prompt calibrated to those tags
Why this order matters: Firing the Nudge Engine after the user's vibe selection — not during the prompt — ensures the AI has access to the user's actual emotional signal before generating the reflection question. A Nudge card for a "golden hour" tag should feel different from one for "city night."

8.3 Trigger Logic

The nudge fires when two or more trigger signals converge. No single signal alone is sufficient — this prevents over-nudging, which is the primary design failure mode for reflection features.

15b.4 Phased Access Architecture

Phase 1 — App Group Shared Container (v1.5, first-party only)

  • Mechanism: niftyMomnt and all first-party companion apps share an iOS App Group container (com.niftymomnt.group). The intelligence graph SQLite database is written by niftyMomnt into this container. Companion apps open the database in read-only mode.
  • Encryption: The SQLite database is encrypted using SQLCipher. The encryption key is derived from the device keychain and shared within the App Group using a secure keychain access group — accessible only to apps signed with the same Apple Developer Team ID.
  • Versioning: The database includes a schema_version integer. niftyMomnt increments this on schema changes. Companion apps check the version on launch and display a "Please update niftyMomnt" prompt if the schema version is ahead of what they support.
  • Offline: Works fully offline. No network required. The shared container is always on-device.
  • Scope: Trip Planner, Annual Review, Travel Portfolio, Shared Journey (all first-party, same Developer Team ID).

Phase 2 — niftyMomnt Swift SDK / XPC (v2.0, third-party)

  • Mechanism: A lightweight Swift SDK (niftyMomntKit) is published to Swift Package Manager. It wraps XPC calls to the niftyMomnt app process, handles per-app user authorization, and returns typed query responses from the intelligence graph. Third-party apps never access the SQLite file directly.
  • Authorization: User grants access per-app in niftyMomnt Settings → Companion Apps. Each app requests a declared set of data types (e.g., "place history and vibe tags only"). Users can revoke at any time.
  • Raw asset access: Third-party apps may request specific raw assets (e.g., the hero Still for a Moment) via the SDK. Each request triggers a user-facing authorization prompt: "[App name] wants to access photos from your Kyoto trip. Allow?" User approves or denies per-request.

Phase 3 — iCloud CloudKit Sync (v2.0+, optional)

  • Mechanism: The intelligence graph (not raw assets) can optionally sync to the user's private iCloud CloudKit database — enabling companion apps on iPad or Mac to access the same data captured on iPhone.
  • Scope: Intelligence graph only. Raw assets remain on the originating iPhone. If a companion app on iPad requests a raw asset, it is fetched on-demand from the iPhone via CloudKit asset transfer — only when the iPhone is reachable and the user is authenticated.
  • Privacy: The CloudKit database uses the user's private iCloud account. niftyMomnt never has access to the CloudKit data server-side — it is user-to-user-device sync only.
  • Opt-in: Off by default. User enables in niftyMomnt Settings → Sync & Backup → "Sync intelligence to iCloud."
